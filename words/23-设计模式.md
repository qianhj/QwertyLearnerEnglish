# 23 设计模式

> GoF 设计模式、架构模式、SOLID 原则相关词汇

| 英文单词 / 短语 | 中文释义 | 场景记忆 |
|---|---|---|
| ⭐ design pattern | 设计模式 | 解决常见设计问题的可复用方案 |
| ⭐ GoF | 四人组 | Gang of Four，设计模式经典著作作者 |
| ⭐ creational pattern | 创建型模式 | 关注对象创建机制的模式 |
| ⭐ structural pattern | 结构型模式 | 关注类/对象组合的模式 |
| ⭐ behavioral pattern | 行为型模式 | 关注对象间通信的模式 |
| ⭐ singleton | 单例模式 | 确保类只有一个实例 |
| ⭐ factory | 工厂模式 | 封装对象创建逻辑 |
| ⭐ factory method | 工厂方法 | 子类决定实例化哪个类 |
| ⭐ abstract factory | 抽象工厂 | 创建相关对象家族 |
| ⭐ builder | 建造者模式 | 分步骤构建复杂对象 |
| ⭐ prototype | 原型模式 | 通过复制创建新对象 |
| ⭐ adapter | 适配器模式 | 接口不兼容类的转换器 |
| ⭐ bridge | 桥接模式 | 抽象与实现分离 |
| ⭐ composite | 组合模式 | 树形结构，统一处理单个和组合对象 |
| ⭐ decorator | 装饰器模式 | 动态给对象添加职责 |
| ⭐ facade | 外观模式 | 为子系统提供统一接口 |
| ⭐ flyweight | 享元模式 | 共享细粒度对象节省内存 |
| ⭐ proxy | 代理模式 | 控制对对象的访问 |
| ⭐ observer | 观察者模式 | 一对多依赖，状态改变通知观察者 |
| ⭐ strategy | 策略模式 | 封装算法族，可互相替换 |
| ⭐ command | 命令模式 | 将请求封装为对象 |
| ⭐ state | 状态模式 | 对象行为随状态改变 |
| ⭐ iterator | 迭代器模式 | 顺序访问聚合对象元素 |
| ⭐ mediator | 中介者模式 | 封装对象间的交互 |
| ⭐ memento | 备忘录模式 | 捕获和恢复对象状态 |
| ⭐ chain of responsibility | 责任链模式 | 多个对象有机会处理请求 |
| ⭐ template method | 模板方法 | 算法骨架，子类实现步骤 |
| ⭐ visitor | 访问者模式 | 分离算法与对象结构 |
| ⭐ MVC | 模型-视图-控制器 | Model-View-Controller 架构 |
| ⭐ MVP | 模型-视图-演示者 | Model-View-Presenter 架构 |
| ⭐ MVVM | 模型-视图-视图模型 | Model-View-ViewModel 架构 |
| ⭐ repository | 仓库模式 | 数据访问抽象层 |
| ⭐ unit of work | 工作单元 | 维护业务对象变更列表 |
| ⭐ dependency injection | 依赖注入 | DI，将依赖从外部传入 |
| ⭐ IoC | 控制反转 | Inversion of Control，框架管理对象生命周期 |
| ⭐ service locator | 服务定位器 | 集中获取依赖的模式 |
| ⭐ lazy loading | 懒加载 | 需要时才创建对象 |
| ⭐ eager loading | 急加载 | 提前加载相关数据 |
| ⭐ SOLID | SOLID原则 | 面向对象设计的五大原则 |
| ⭐ SRP | 单一职责原则 | Single Responsibility Principle |
| ⭐ OCP | 开闭原则 | Open/Closed Principle，对扩展开放对修改关闭 |
| ⭐ LSP | 里氏替换原则 | Liskov Substitution Principle |
| ⭐ ISP | 接口隔离原则 | Interface Segregation Principle |
| ⭐ DIP | 依赖倒置原则 | Dependency Inversion Principle |
| ⭐ DRY | 不要重复自己 | Don't Repeat Yourself |
| ⭐ KISS | 保持简单 | Keep It Simple, Stupid |
| ⭐ YAGNI | 你不需要它 | You Ain't Gonna Need It |
| ⭐ composition | 组合 | 优于继承的设计方式 |
| ⭐ inheritance | 继承 | 类之间的层次关系 |
| ⭐ polymorphism | 多态 | 同一接口不同实现 |
| ⭐ encapsulation | 封装 | 隐藏内部实现细节 |
| ⭐ abstraction | 抽象 | 提取共同特征 |
| ⭐ interface | 接口 | 定义契约，不包含实现 |
| ⭐ abstract class | 抽象类 | 不能实例化的基类 |
| ⭐ concrete class | 具体类 | 可实例化的类 |
| ⭐ implementation | 实现 | 接口的具体代码 |
| ⭐ inheritance | 继承 | 子类获得父类特性 |
| ⭐ delegation | 委托 | 将任务交给其他对象 |
| ⭐ aggregation | 聚合 | 整体-部分关系，部分可独立存在 |
| ⭐ composition | 组合 | 整体-部分关系，部分不能独立存在 |
| ⭐ association | 关联 | 对象间的连接关系 |
| ⭐ coupling | 耦合 | 模块间的依赖程度 |
| ⭐ loose coupling | 松耦合 | 模块间依赖少，易维护 |
| ⭐ tight coupling | 紧耦合 | 模块间依赖多，难维护 |
| ⭐ cohesion | 内聚 | 模块内部元素的关联程度 |
| ⭐ high cohesion | 高内聚 | 模块职责单一明确 |
| ⭐ separation of concerns | 关注点分离 | 不同功能模块化分离 |
| ⭐ cross-cutting concern | 横切关注点 | 影响多模块的功能（日志、安全等）|
| ⭐ AOP | 面向切面编程 | Aspect-Oriented Programming |
| ⭐ aspect | 切面 | 横切关注点的模块化 |
| ⭐ join point | 连接点 | 程序执行中的特定点 |
| ⭐ pointcut | 切入点 | 匹配连接点的表达式 |
| ⭐ advice | 通知 | 切面在连接点执行的动作 |
| ⭐ anti-pattern | 反模式 | 常见但不良的设计方案 |
| ⭐ god class | 上帝类 | 做太多事情的臃肿类 |
| ⭐ spaghetti code | 面条代码 | 结构混乱难以维护的代码 |
| ⭐ blob | 泥球 | 把所有东西放在一起的反模式 |
