# 22 测试与 QA

> 软件测试、自动化测试、质量保障相关词汇

| 英文单词 / 短语 | 中文释义 | 场景记忆 |
|---|---|---|
| ⭐ QA | 质量保证 | Quality Assurance，确保软件质量的系统性活动 |
| ⭐ testing | 测试 | 验证软件是否满足需求 |
| ⭐ test case | 测试用例 | 包含输入、条件、预期结果的测试规格说明 |
| ⭐ test suite | 测试套件 | 一组相关的测试用例 |
| ⭐ test plan | 测试计划 | 描述测试策略、资源、进度的文档 |
| ⭐ test scenario | 测试场景 | 高级功能描述，可包含多个测试用例 |
| ⭐ unit test | 单元测试 | 测试最小的代码单元（函数、方法）|
| ⭐ integration test | 集成测试 | 测试多个模块/服务间的交互 |
| ⭐ system test | 系统测试 | 测试完整系统的功能和性能 |
| ⭐ acceptance test | 验收测试 | 确认系统是否满足业务需求，决定是否发布 |
| ⭐ UAT | 用户验收测试 | User Acceptance Testing |
| ⭐ regression test | 回归测试 | 修改后验证原有功能是否正常 |
| ⭐ smoke test | 冒烟测试 | 快速验证基本功能，决定是否继续深入测试 |
| ⭐ sanity test | 健全性测试 | 快速检查新构建是否可用 |
| ⭐ functional test | 功能测试 | 验证功能是否符合需求 |
| ⭐ non-functional test | 非功能测试 | 性能、安全、可用性等测试 |
| ⭐ black box test | 黑盒测试 | 不了解内部代码，只测输入输出 |
| ⭐ white box test | 白盒测试 | 了解内部代码结构进行测试 |
| ⭐ gray box test | 灰盒测试 | 部分了解内部结构的测试 |
| ⭐ manual test | 手工测试 | 人工执行的测试 |
| ⭐ automated test | 自动化测试 | 使用脚本和工具自动执行的测试 |
| ⭐ test script | 测试脚本 | 自动化测试的代码脚本 |
| ⭐ test framework | 测试框架 | Jest、JUnit、Pytest 等测试工具 |
| ⭐ assertion | 断言 | 验证预期结果的声明 |
| ⭐ mock | 模拟对象 | 模拟依赖行为的假对象 |
| ⭐ stub | 存根 | 返回预设响应的简单模拟 |
| ⭐ spy | 间谍 | 记录调用信息的包装对象 |
| ⭐ fixture | 测试夹具 | 测试前的准备数据和状态 |
| ⭐ setup | 前置准备 | 测试前的初始化操作 |
| ⭐ teardown | 后置清理 | 测试后的清理操作 |
| ⭐ test coverage | 测试覆盖率 | 代码/需求被测试的比例 |
| ⭐ code coverage | 代码覆盖率 | 被测试执行到的代码比例 |
| ⭐ line coverage | 行覆盖率 | 被测试覆盖的代码行比例 |
| ⭐ branch coverage | 分支覆盖率 | 被测试覆盖的代码分支比例 |
| ⭐ TDD | 测试驱动开发 | Test-Driven Development，先写测试再写代码 |
| ⭐ BDD | 行为驱动开发 | Behavior-Driven Development，用自然语言描述测试 |
| ⭐ given-when-then | 给定-当-那么 | BDD 测试描述格式 |
| ⭐ CI/CD | 持续集成/部署 | Continuous Integration / Continuous Deployment |
| ⭐ test environment | 测试环境 | 执行测试的软硬件环境 |
| ⭐ staging | 预发布环境 | 接近生产环境的测试环境 |
| ⭐ production | 生产环境 | 正式运行的环境 |
| ⭐ bug / defect | 缺陷 | 软件中的错误 |
| ⭐ issue | 问题 | 泛指 bug、需求、任务等 |
| ⭐ severity | 严重级别 | 缺陷对功能的影响程度 |
| ⭐ priority | 优先级 | 修复缺陷的紧急程度 |
| ⭐ blocker | 阻塞性 | 阻止发布的严重问题 |
| ⭐ critical | 严重 | 核心功能不可用 |
| ⭐ major | 主要 | 重要功能有问题 |
| ⭐ minor | 次要 | 非重要功能问题 |
| ⭐ trivial | 轻微 | 很小的问题，如拼写错误 |
| ⭐ bug report | 缺陷报告 | 描述缺陷的文档 |
| ⭐ bug tracker | 缺陷跟踪系统 | Jira、Bugzilla 等 |
| ⭐ reproduce | 复现 | 重现缺陷的步骤 |
| ⭐ steps to reproduce | 复现步骤 | 导致缺陷的操作序列 |
| ⭐ expected result | 预期结果 | 测试应该得到的结果 |
| ⭐ actual result | 实际结果 | 测试实际得到的结果 |
| ⭐ boundary value | 边界值 | 输入范围的边界数据 |
| ⭐ equivalence class | 等价类 | 相同处理方式的数据分组 |
| ⭐ edge case | 边界情况 | 极端或异常的输入条件 |
| ⭐ corner case | 角落情况 | 多种边界条件同时发生的情况 |
| ⭐ happy path | 正常路径 | 一切正常的最理想执行路径 |
| ⭐ unhappy path | 异常路径 | 出现错误时的执行路径 |
| ⭐ positive test | 正向测试 | 验证正常输入的处理 |
| ⭐ negative test | 负向测试 | 验证异常输入的处理 |
| ⭐ load test | 负载测试 | 测试系统在正常和峰值负载下的表现 |
| ⭐ stress test | 压力测试 | 测试系统在极限条件下的稳定性 |
| ⭐ performance test | 性能测试 | 测试响应时间、吞吐量等指标 |
| ⭐ benchmark | 基准测试 | 测量性能的对比测试 |
| ⭐ soak test | 浸泡测试 | 长时间运行测试稳定性 |
| ⭐ spike test | 尖峰测试 | 突然增加负载测试系统反应 |
| ⭐ concurrency test | 并发测试 | 多用户同时操作的测试 |
| ⭐ usability test | 可用性测试 | 测试用户体验和易用性 |
| ⭐ accessibility test | 无障碍测试 | 测试残障人士使用便利性 |
| ⭐ compatibility test | 兼容性测试 | 不同环境/浏览器/设备的测试 |
| ⭐ cross-browser test | 跨浏览器测试 | 不同浏览器的兼容性测试 |
| ⭐ E2E test | 端到端测试 | End-to-End，完整用户流程测试 |
| ⭐ API test | API测试 | 接口级别的测试 |
| ⭐ contract test | 契约测试 | 验证服务间接口契约 |
| ⭐ visual test | 视觉测试 | 验证 UI 视觉效果 |
| ⭐ snapshot test | 快照测试 | 对比 UI 或数据结构快照 |
| ⭐ flaky test | 不稳定测试 | 有时通过有时失败的测试 |
| ⭐ false positive | 假阳性 | 实际无缺陷但被报告 |
| ⭐ false negative | 假阴性 | 实际有缺陷但未发现 |
| ⭐ build | 构建 | 编译打包后的软件版本 |
| ⭐ release | 发布 | 正式推出的软件版本 |
| ⭐ version | 版本 | 软件的特定发布 |
| ⭐ regression | 回归 | 修复或新功能引入的旧问题 |
| ⭐ alpha | 内测版 | 内部测试的早期版本 |
| ⭐ beta | 公测版 | 外部用户测试的版本 |
| ⭐ RC | 候选发布版 | Release Candidate |
| ⭐ GA | 正式版 | General Availability |
| ⭐ hotfix | 热修复 | 紧急修复生产问题 |
| ⭐ patch | 补丁 | 修复问题的更新 |
| ⭐ test data | 测试数据 | 执行测试使用的数据 |
| ⭐ seed data | 种子数据 | 初始化环境的基础数据 |
| ⭐ test double | 测试替身 | 替代真实依赖的对象 |
| ⭐ SUT | 被测系统 | System Under Test |
| ⭐ DUT | 被测设备 | Device Under Test |
| ⭐ QA engineer | 测试工程师 | 负责质量保证的工程师 |
| ⭐ SDET | 软件测试开发工程师 | Software Development Engineer in Test |
